代码分离

代码分离是 webpack 中最引人注目的特性之一。此特性能够把代码分离到不同的 bundle 中，然后可以按需加载或并行加载这些文件。代码分离可以用于获取更小的 bundle，以及控制资源加载优先级，如果使用合理，会极大影响加载时间。

有三种常用的代码分离方法：

  • 入口起点：使用 entry 配置手动地分离代码。这个不用说.
  • 防止重复：使用 webpack内置的CommonsChunkPlugin 去重和分离 chunk。
  • 动态导入：通过模块的内联函数调用来分离代码。

CommonsChunkPlugin 插件可以将公共的依赖模块提取到已有的入口 chunk 中，或者提取到一个新生成的 chunk。

基础概念

  • chunk: webpack将多个模块打包之后的代码集合,entry:{index:'index.js',math:'math.js'}index和math就是chunk
  • Runtime: 非源码部分的引导模块代码，都属于 webpack 运行时。(用了CommonsChunkPlugin,可以简单看下源码,就明白了)
  • normal chunk: 普通块不包含运行时runtime，只包含一系列模块。但是在应用运行时，普通块可以动态的进行加载。通常会以jsonp的包装方式进行加载。而code splitting主要使用的就是普通块。
  • initial chunk: 官方对initial chunk的定义非常简单，初始块就是普通块，跟普通块相同的是同样不包含运行时runtime，不同的是初始块是计算在初始加载过程时间内的。在介绍入口块entry chunk的时候也介绍过，一旦入口块失去了运行时，就会变成初始块。这个转变经常由CommonsChunkPlugin插件实现。
  • Entry Chunk: 包含一系列模块代码，以及webpack的运行时(Runtime)代码，一个页面只能有一个Entry Chunk，并且需要先于Normal Chunk载入
  • CommonsChunkPlugin的本质: 将多个入口中的公有代码和Runtime(运行时)抽取到父节点, 如果是单入口文件则会将Runtime(运行时)抽取出来

可以这样理解,没有CommonsChunkPlugin,各入口生成的文件互不相干
有了CommonsChunkPlugin,将各入口(chunks的配置项的,没有chunks默认是所有)生成的文件Runtime和共同部分扔出来了,扔到哪个文件,就看配置了,有可能放在哪个入口文件里,有可能新建一个文件

  • index bundle 会随着自身的新增内容的修改，而发生变化。
  • vendor bundle 会随着自身的 module.id 的修改，而发生变化。
  • runtime bundle 会因为当前包含一个新模块的引用，而发生变化。

项目的js规范可能各有千秋， 而eslint可以自定义规范.更重要的是， 我想要编辑器根据此规范自动格式化js代码。

安装

  • 全局安装npm install eslint -g
  • 推荐本地项目安装npm install eslint --save-dev，在运行此命令之前， 项目必须有package.json文件，可以用npm init --y快速生成(以下按照本地安装)
  • 生成配置文件 ./node_modules/.bin/eslint --init，一般在第一个问题的时候选择use a popular style guide，然后选择standard，这样配置文件.eslintrc.js就生成了
  • 检测并格式化某js文件 ./node_modules/.bin/eslint youfile.js --fix 不加fix只是显示错误并不纠正

配置vscode

我用的是vscode，所以说下vscode配置eslint，保存的时候自动按照配置文件格式化。

1.vscode下载插件eslint
2.文件 -> 首选项 -> 设置 -> 搜索eslint， 右侧配置

具体配置参数

常用:

  • Environments - 指定脚本的运行环境。每种环境都有一组特定的预定义全局变量。
  • Globals - 脚本在执行期间访问的额外的全局变量
  • Rules - 启用的规则及各自的错误级别.
    • “off” 或 0 - 关闭规则
    • “warn” 或 1 - 开启规则，使用警告级别的错误：warn (不会导致程序退出)
    • “error” 或 2 - 开启规则，使用错误级别的错误：error (当被触发的时候，程序会退出)
  • “eslint:recommended” 的 extends 属性启用一系列核心规则
  • 特殊禁止规则
    • /* eslint-disable */ area code /* eslint-enable */
    • // eslint-disable-next-line

不常用:

  • parserOptions - 属性设置解析器选项
  • plugins - ESLint 支持使用第三方插件。在使用插件之前，你必须使用 npm 安装它。

editorconfig让代码美化更加省心

Posted on 2017-11-20 | In 编辑器 |

简介

官网网址
EditorConfig让编辑器读取配置文件并依此格式化代码。
比如保存的时候,编辑器会自动在文件末尾添加新的一行.
有些编辑器会默认支持, 有些需要安装响应的插件, 官网有说明.

就近原则

当打开一个文件时，EditorConfig插件会在打开文件的目录和其每一级父目录查找.editorconfig文件，直到有一个配置文件root=true。
EditorConfig配置文件从上往下读取，并且路径最近的文件最后被读取。匹配的配置属性按照属性应用在代码上，所以最接近代码文件的属性优先级最高。一般放在项目根目录。

配置文件的名字是.editorconfig

支持的属性

  • indent_style：tab为hard-tabs，space为soft-tabs。
  • indent_size：设置整数表示规定每级缩进的列数和soft-tabs的宽度（译注：空格数）。如果设定为tab，则会使用tab_width的值（如果已指定）。
  • tab_width：设置整数用于指定替代tab的列数。默认值就是indent_size的值，一般无需指定。
  • end_of_line：定义换行符，支持lf、cr和crlf。
  • charset：编码格式，支持latin1、utf-8、utf-8-bom、utf-16be和utf-16le，不建议使用uft-8-bom。
  • trim_trailing_whitespace：设为true表示会除去换行行首的任意空白字符，false反之。
  • insert_final_newline：设为true表明使文件以一个空白行结尾，false反之。
  • root：表明是最顶层的配置文件，发现设为true时，才会停止查找.editorconfig文件。

理解async

Posted on 2017-10-28 | In js |

TL;DR

  • promise 对象最重要的两个属性,状态和值
  • async 函数返回promise对象, 函数不出错的话默认就是resolved状态, return的值就是promise的值
  • await 必须在async函数中, 后面是promise对象, 返回值是promise对象的值
  • await 必须等到promise的状态改变, 才会执行下一条语句

先回忆下promise

  • promise对象有两个属性, 状态和值,初始状态和值都是undefined
  • 一旦建立, 立即执行
  • 执行遇到resolve,状态就是resolved,值是resolve的第一个参数,reject同理

上例中,timeout(500,'hello')就表示此promise对象的状态是resolved, 值是hello

async函数

  • async函数返回一个 Promise 对象。
  • 返回的promise对象的状态默认是resolved, 值是函数的返回值
  • 特殊: async函数只有内部抛出错误的时候, 这个Promise对象的状态是rejected,值是错误对象

上例中, t就是promise对象, 状态是resolved, 值是 我就是值

await语句

  • await 命令只能用在 async 函数内部，如果用在普通函数，就会报错
  • 正常情况下，await命令后面是一个 Promise 对象。如果不是，会被转成一个立即resolve的 Promise 对象。类似执行了Promise.resolve()
  • await的返回值是promise对象的值
  • 特殊情况下, 如果await后面的promise对象的状态是rejected, 那么会报错, 所以通常async函数执行放在try,catch里, 一旦报错就会走catch
